# Biohacking — Personal Research Knowledge Base

A structured map of the field called "biohacking," built to answer three questions:

1. **What is this actually?** The word is a bucket covering six-plus distinct scenes with different tools, cultures, risk profiles, and evidence standards. Most confusion comes from collapsing them.
2. **What's substantive vs. marketing noise?** Some corners are peer-reviewed science or serious engineering communities. Others are supplement funnels wearing lab coats. This repo tags the difference.
3. **Is it worth deeper engagement?** Answered over time in `fit-assessment.md`, using notes accumulated here.

Audience assumption: a senior engineer with a security background who wants signal density, not encouragement.

## Ground rules

These apply to every file, and to future-me adding to it.

- **No shopping.** This is a research KB, not a buying guide. Where a product is unavoidable context (implant chips, CGMs, EEG headsets), describe it as a technical/cultural artifact, not a recommendation. No affiliate links, no "best of" ranking.
- **Distinguish primary from secondary.** Primary = the person/org's own writing, a peer-reviewed paper, a foundational talk, source code, a community wiki maintained by practitioners. Secondary = journalism, explainers, third-party summaries. Every reading/viewing list separates them.
- **Note dates.** Fields move. A 2014 claim about implant capabilities or longevity science is a historical data point, not a current-state claim. Date things where it matters.
- **Flag commerce.** Content that exists to sell a supplement, protocol, subscription, device, or personal brand gets a **`[commerce]`** tag inline, regardless of whether it's also interesting. The tag is not a verdict — it's a disclosure so it doesn't get mixed up with peer-reviewed or community-primary material.
- **Use "biohacking" only with a subdomain tag.** The bare word is nearly meaningless here. Say "DIY bio," "grinder/implantable," "quantified self," etc. When you must use the umbrella term, pair it with which scene you mean.

## Confidence conventions

- **`[verify]`** — claim I'm not certain of (whether a person is still active, whether an org still operates, whether a link still resolves, a specific date). Treat as a lead to confirm, not a fact. Where these appear, I chose to flag rather than assert.
- **`[primary]` / `[secondary]`** — source type, used in reading lists.
- **`[commerce]`** — has a commercial/sales motive; see ground rules.
- No sources are fabricated. Where I don't have a link, I name the thing and leave finding the canonical URL as a `[verify]`.

## How to use it

- Start with `00` to get the taxonomy straight, then read the subdomain files that match your actual interest.
- Each subdomain file follows the same skeleton: What it is → Key figures → Key orgs/projects → Foundational reading → Current state (2026) → Red flags → Adjacent scenes.
- `09` and `10` are indexes — use them to jump laterally.
- Keep opinions in `fit-assessment.md`, keep facts in the numbered files.

## Scope note

This is field research on biohacking as a domain. It deliberately does not cover any specific conference, meetup, or event. If you want the "where do people gather" angle, that lives in the community/org sections of each subdomain file, keyed to standing institutions (labs, foundations, open-source projects), not calendar events.
